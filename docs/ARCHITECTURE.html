Cairn · v1 build plan

A read-only canonical projection of a claim graph.

Agentic-AI-ready-first. An AI coding agent authors claims — conclusions grounded in evidence — into a local, file-as-truth store during normal analysis work; humans and fresh AI sessions read a published, immutable projection. This is the architecture I'll build.

Core: Bun + TypeScript Index: bun:sqlite (derived, in-memory) Site: React on vinext Store: cairn/ subdir in host repo Truth: plain-text claim files in git

02 · The three hard rules

Non-negotiable. Everything else is swappable.
1

Every claim must reach the ground

A claim has ≥1 edge, and following depends_on edges upward must terminate at a real artifact — never resting only on other claims (that's circular reasoning in a provenance costume). Enforced by a recursive SQL reach-ground query at the draft→canonical promotion gate, not at write time. Drafts may be ungrounded while you work; nothing ungrounded is ever shared.

2

Freshness is computed from the evidence fingerprint, never typed

fresh / stale / unknown comes from comparing the artifact's fingerprint now to what was stamped at authoring — never hand-set, never AI-guessed, never a field in the file. Quality is tiered and shown on the badge. unknown (artifact unreachable) is a legal, honest state. A false fresh is the enemy.

fresh · pipeline-meta stale · sha256 unknown · remote-md5 unverified
3

Each publish is an immutable snapshot + diff

The head moves forward; reruns cascade staleness; a reader sees exactly what changed since the snapshot they last saw ("Since <prev>: N changed"). Past snapshots are never mutated — content-addressed by a hash of their head data.

canonical ≠ verified. Publishing makes a version the agreed current record, never true. Verification is a separate axis; v1 stores an honest unverified default only — the machinery is v2.

03 · The claim file

Source of truth — deliberately no freshness field
--- cairn/claims/claim-20260610-001.md ---
id: claim-20260610-001
text: "Step-07 scoring separates the two cohorts (AUC 0.91)"
status: draft            # draft | canonical  (draft may be ungrounded)
verification: unverified  # v1: honest default only
grounding:               # claim → evidence  (≥1 required to leave draft)
  - kind: target        # target | file | data | external
    ref: results_step_07
    fingerprint: 90e13daf5941a99d
    method: pipeline-meta # top tier — read from _targets/meta/meta
  - kind: file
    ref: outputs/step07_scores.csv
    fingerprint: "sha256:…"
    method: sha256
depends_on:              # claim → claim; does NOT count as grounding
  - claim-20260609-014
created_at: 2026-06-10T20:00:00-04:00
---
Optional freeform notes / caveats. Not parsed in v1.
Fingerprint tiers (honest about rigor): a pipeline tool's content hash (targets' _targets/meta/meta) is top tier and free; a direct sha256 of a loose file is mid; a self-reported remote md5sum is lower; an unreachable artifact reads unknown. The tier is always shown — never flattened.

04 · The skill — the agent's 4-touchpoint loop

The only authoring driver in v1

① Orient — at session start

Read canonical state + pending drafts before acting. This is what lets a fresh session land oriented.

cairn head

② Author — the moment you conclude anything

One cheap, in-flow call. Draft-soft, so there's no friction tax that breeds avoidance. Ground it a moment later if needed.

cairn add-claim --text "…" --evidence file:outputs/scores.csv

③ Refresh — after any rerun

Recompute freshness, surface newly-stale claims to the user.

cairn refresh

④ Publish — before sharing

Reach-ground gate, then freeze a snapshot + diff. Relays the warn-only reconcile honestly.

cairn validate && cairn publish
Enforcement model (v1): make lapses visible, don't pretend they were prevented. publish runs a warn-only reconcile — it reports conclusions in the shared findings that carry no claim id, and drafts left ungrounded, but does not block. Claude Code hooks that nudge without relying on goodwill are the v2 hardening.

05 · The published projection — two readers, one store

React site on vinext, emitted as static assets inside every snapshot
READER A

Collaborator

A static, read-only page. No login, just a link — friction below "open a zip." Each claim shows its text, terse grounding refs, an honest freshness badge (state + tier) and verification badge, with the diff-since-prev banner at the top. Portable enough to later serve from a Cloudflare Worker unchanged.

READER B

Fresh agent session

head.json — a compact, typed snapshot of the canonical head: each claim's id/text/verification/derived-freshness/edges, plus open drafts. Selected state, not a transcript dump. The acceptance test feeds only this to a new session and asks "where are we / what next."

Frontend stack & taste: React on vinext (Cloudflare's Vite-based Next.js reimplementation) building to fully static assets that fetch relative ./data/head.json + ./data/diff.json and render client-side. Self-hosted Inter variable, generous whitespace, strong type hierarchy, light/dark via prefers-color-scheme. Motion: staggered card entrance, spring hover micro-interactions, View Transitions, scroll-driven reveals — all gated behind prefers-reduced-motion. vinext is <1 week old & experimental; if it blocks static output, I fall back to plain Vite + React with the same components and tell you.

06 · Fingerprint methods & freshness cascade

How "did it change?" is answered, by tier
MethodSourceTierRe-check at refresh
pipeline-metatargets _targets/meta/meta data columnTop (rigorous, free)Re-read meta row, compare hash
sha256Direct hash of a loose local fileMidRe-hash file at location
size-mtimeWeak fallback when hashing is too costlyLowRe-stat, compare size+mtime
remote-md5Agent runs md5sum on remote HPC in-sessionLow (self-reported)ssh host md5sum; unreachable → unknown
Cascade precedence: a claim is stale if any grounding edge changed or any depends_on is stale; else unknown if any edge is unknown; else fresh. Stale wins over unknown. Traversal is cycle-safe.

08 · How I'll build it

One orchestrated workflow, with verification gates
PHASE 1

Scaffold + contracts

Pin docs/CONTRACTS.md + src/types.ts: claim format, store layout, head.json/diff.json/snapshot schemas, CLI signatures, promotion semantics. Everything downstream codes against this.

PHASE 2

Three parallel tracks

Core library + CLI · the vinext React site · the skill. Independent, built concurrently against the frozen contracts.

PHASE 3

Integrate

Wire publish→site bundling, build the demo fixture, write the full-loop acceptance script (add 3 claims → publish → mutate artifact → refresh → re-publish → assert staleness + immutability + diff).

PHASE 4–5

Verify & fix

Independent functional verify + code review + browser site QA in parallel; fix blocking findings; re-verify (≤2 rounds). The three BUILD-BRIEF failure conditions must be proven impossible.

Failure conditions that must stay impossible: an ungrounded claim reaching canonical · freshness stored as a column · artifacts copied into the store · a claim that can't reach ground getting published.
